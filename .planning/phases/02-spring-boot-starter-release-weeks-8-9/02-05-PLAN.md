<objective>
Close the two BLOCKER verification gaps (CR-01, CR-02) found in Phase 2 so the
budget-breaker Spring Boot starter is safe for the full intended consumer base and
does not silently lose its primary safety signal (token/cost/breach metrics).

Gap 1 (CR-01): The auto-config references Actuator/Micrometer types directly in @Bean
method signatures with those dependencies declared `compileOnly`. A consumer Spring Boot
app without `spring-boot-actuator-autoconfigure` or `micrometer-core` on its runtime
classpath fails context refresh with NoClassDefFoundError. Fix by splitting the
Actuator and Micrometer beans into nested @Configuration classes guarded by
@ConditionalOnClass — the established Spring Boot starter idiom.

Gap 2 (CR-02): Both MetricsEventCollector and SLF4JEventLogger run
`breaker.events.collect { ... }` with no try/catch inside the lambda. The first handler
exception terminates the collect call permanently while isRunning() still reports true,
silently killing all future event handling. Fix by wrapping the when(event) dispatch in
try/catch that logs at WARN, and correct the KDoc that falsely credits SupervisorJob.

Purpose: Restore SPRING-01 (optional-dependency design intent) and SPRING-03 (reliable
Micrometer metrics) to a SATISFIED state so the v0.1.0 starter is usable and dependable.
Output: Modified auto-config + both collectors, plus new FilteredClassLoader and resilience tests.
</objective>
